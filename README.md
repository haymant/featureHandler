# Alpha repro package

This directory contains a minimal subset of **qlib** required to
recalculate the Alpha158/Alpha360 feature sets.  It is self‑contained
(except for `qlib`'s data files) and is meant to be copy‑pasteable into
new projects for experimentation or lightweight benchmarking.

## Cython prerequisites

Alpha158 performance depends on a small Cython extension providing
rolling statistics (`featureHandler/_libs/rolling`).  When you clone or
move this folder to a new location the compiled module does **not** travel
with it; you must rebuild it before you run the script again.

The build step compiles against your local **numpy** and **Cython**
versions.  Mismatches can trigger compilation errors such as
```
error: ‘PyArray_Descr’ has no member named ‘subarray’
```
which indicate an incompatible NumPy API.  To avoid this:

   **Supporting newer NumPy releases:**

   The underlying problem is that the C++ generated by Cython is tied to
   a specific NumPy ABI.  When NumPy introduces breaking changes (as
   happened in versions after 1.25) the generated code may refer to
   struct members that no longer exist.  Rebuilding with a newer Cython
   release typically solves this because the regenerated code uses the
   updated API; hence always use the latest Cython you can install.
   If upgrading Cython is not possible, you can manually copy a prebuilt
   `rolling` shared object from a compatible qlib install, or take the
   fallback route described below.

   To summarise:
1. upgrade Cython
   (e.g. `uv add --upgrade cython` or `pip install -U cython`).
      After installation the next build of the extension will regenerate
      `rolling.cpp` automatically, so you *do not* need to delete the old
      file by hand – the `build_ext.py` script overwrites it.  Removing
      the stale `.cpp` is harmless however and can be done if you want to
      be extra cautious.
2. run the build command again (`uv run build_ext.py build_ext --inplace`)
   from the `featureHandler` directory.  The `NPY_NO_DEPRECATED_API` macro
   in `build_ext.py` helps with warnings, but it cannot magically adapt
   to ABI breaks.

   If you still see the `PyArray_Descr has no member 'subarray'` error even
   after updating Cython, it means the NumPy API has changed in a way that
   the current `rolling.pyx` generator does not understand.  fixing this
   requires one of:

   * edit `rolling.pyx` to remove all usages of NumPy dtype internals
     (avoiding `np.ndarray` or `np.PyArray_Descr` in Cython) and implement
     the minimal rolling algorithms manually in C; or
   * manually patch the generated `rolling.cpp` to delete the two
     references to `->subarray` (they occur in the dtype shape helper and
     are harmless for our purposes); after patching you can compile with
     any NumPy version.
   * or, simplest, copy a prebuilt binary from a working qlib `data/_libs`
     directory as described above.

   In practice, upgrading Cython usually solves the problem; rewriting the
   extension should only be attempted if you plan to maintain compatibility
   with cutting‑edge NumPy releases on an ongoing basis.


1. Use a recent Cython release and rebuild. Compatibility is driven
   by the Cython-generated code, not by a fixed NumPy version; upgrading
   Cython is usually sufficient to support the latest NumPy API.
2. Avoid blindly mixing very old NumPy headers with a new Cython.
   Start from a fresh virtualenv and install the requirements listed in
   `pyproject.toml` before building.  
3. The provided `build_ext.py` sets `NPY_NO_DEPRECATED_API` to suppress
   deprecation warnings, but it cannot fix ABI changes automatically.

If you still cannot compile at all, set `USE_CYTHON=False` in
`featureHandler/provider.py` to fall back to the slower pure-Python
loader (Alpha158 will then take ~10 s).  Alternatively, install full
qlib and import its provider directly; that is the simplest way to get
fast behaviour without local compilation.

Steps to prepare:

1. Install a Python environment with at least:
   ```sh
   uv venv
   uv init
   uv add Cython numpy pandas setuptools
   ```
   (qlib itself is not required unless you want to compare outputs)

2. Compile the extension.  From the package root (`scripts/Alpha`):
   ```sh
   cd scripts/Alpha/featureHandler
   uv run build_ext.py build_ext --inplace
   ```
   This will produce `featureHandler/_libs/rolling*.so` (or `.pyd` on
   Windows).  The top‑level `__init__` already contains a helper that
   runs this command automatically when you import `featureHandler`, but
   the manual step ensures the build succeeds before any other code runs.

3. (Optional) if you don't have Cython or cannot compile, set
   `USE_CYTHON=False` in `featureHandler/provider.py`.  The pure‑Python
   provider will still work but Alpha158 will take 31016s
   instead of ~2s.

## Running the benchmark

```sh
cd scripts/Alpha
uv run ./calc158.py --start 2008-01-01 --end 2020-12-31 --inst AAPL
```

Expected timings with the extension built: ~2.2s for Alpha158, ~2.2s
for Alpha360.  If the extension is missing the script will fall back to
pure Python and take ~10s+.

## Moving this directory

When copying `scripts/Alpha` to another repository or location, also
copy the `featureHandler` subdirectory.  Once copied:

* recreate the virtual environment or install the prerequisites
* run the build step described above
* run `uv run ./calc158.py` as usual

The rest of the code has no external dependencies beyond standard
scientific Python libraries.
